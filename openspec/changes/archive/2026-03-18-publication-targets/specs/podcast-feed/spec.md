## MODIFIED Requirements

### Requirement: Feed metadata includes podcast and user name
The RSS feed title SHALL use the podcast name only (e.g., `"{podcastName}"`).

#### Scenario: Feed title with podcast name
- **WHEN** the RSS feed is generated for podcast "The Weekly JVM Podcast"
- **THEN** the feed title is "The Weekly JVM Podcast"

### Requirement: RSS 2.0 podcast feed generation
The system SHALL generate a valid RSS 2.0 XML file using ROME's `SyndFeed` API. The feed SHALL include standard podcast metadata (title, description, link) from application configuration, and each episode SHALL be represented as an `<item>` with an `<enclosure>` tag pointing to the MP3 file URL. The feed SHALL be regenerated each time a new episode is produced. The feed SHALL include a `<language>` element set to the podcast's configured language code. When the podcast has a stored image file, the feed SHALL include an `<image>` element with `<url>`, `<title>`, and `<link>` sub-elements. The image URL SHALL use the effective base URL (e.g., `{baseUrl}/episodes/{podcastId}/podcast-image.jpg`).

#### Scenario: Feed generated with episodes
- **WHEN** 3 episodes exist in the database
- **THEN** the RSS feed contains 3 `<item>` entries, each with a valid `<enclosure>` tag containing the MP3 URL, file size in bytes, and MIME type `audio/mpeg`

#### Scenario: Feed generated with no episodes
- **WHEN** no episodes exist in the database
- **THEN** the RSS feed is valid XML with channel metadata but no `<item>` entries

#### Scenario: Feed updated after new episode
- **WHEN** a new episode is generated by the TTS pipeline
- **THEN** the RSS feed XML is regenerated to include the new episode

#### Scenario: Feed includes language element
- **WHEN** the RSS feed is generated for a podcast with language `"nl"`
- **THEN** the RSS XML contains `<language>nl</language>` in the channel element

#### Scenario: Feed language defaults to English
- **WHEN** the RSS feed is generated for a podcast with default language `"en"`
- **THEN** the RSS XML contains `<language>en</language>` in the channel element

#### Scenario: Feed includes podcast image
- **WHEN** the RSS feed is generated for a podcast that has a stored `podcast-image.jpg`
- **THEN** the RSS XML contains an `<image>` element with `<url>{baseUrl}/episodes/{podcastId}/podcast-image.jpg</url>`, `<title>{podcastName}</title>`, and `<link>{baseUrl}</link>`

#### Scenario: Feed without podcast image
- **WHEN** the RSS feed is generated for a podcast that has no stored image
- **THEN** the RSS XML does not contain an `<image>` element

## ADDED Requirements

### Requirement: Episode description uses recap only with sources link
Episode `<description>` in the RSS feed SHALL contain only the recap text. If a `publicUrl` is available (from FTP target config or static base URL), the description SHALL also include a link to the episode's `sources.md` file: `"{recap}\n\nSources: {publicUrl}/{slug}-sources.md"`. If no `publicUrl` is available, the description SHALL contain only the recap text. The description SHALL NOT contain inline source listings.

#### Scenario: Description with publicUrl available
- **WHEN** the feed is generated for a podcast with an FTP target configured with `publicUrl = "https://podcast.example.com/shows/tech/"` and an episode with recap and sources
- **THEN** the episode description is `"{recap}\n\nSources: https://podcast.example.com/shows/tech/tech-news-2026-03-09-sources.md"`

#### Scenario: Description without publicUrl
- **WHEN** the feed is generated for a podcast with no FTP target configured and an episode with recap
- **THEN** the episode description is just the recap text (no sources link, no inline sources)

#### Scenario: Description without recap
- **WHEN** the feed is generated for an episode without a recap
- **THEN** the episode description falls back to the first 500 characters of the script text + "..."

### Requirement: Show notes format change
The `showNotes` field on episodes SHALL contain only the recap text (no inline source list). The `buildShowNotes` method SHALL no longer append the "Sources:" section with article titles and URLs. The sources information is captured in the separate `sources.md` file instead.

#### Scenario: New episode show notes
- **WHEN** a new episode is created with a recap and linked articles
- **THEN** the `showNotes` field contains only the recap text

#### Scenario: Show notes without recap
- **WHEN** a new episode is created without a recap
- **THEN** the `showNotes` field is null

### Requirement: Regenerate existing episode show notes
The system SHALL provide a mechanism to regenerate `showNotes` for all existing episodes to the new recap-only format. For each existing episode that has a `recap` field, the `showNotes` SHALL be updated to contain only the recap text (removing any inline source listings). Episodes without a `recap` field SHALL keep their existing `showNotes` unchanged.

#### Scenario: Episode with recap and old-format show notes
- **WHEN** regeneration runs for an episode with `recap = "Summary text"` and `showNotes = "Summary text\n\nSources:\n- Article 1\n  https://..."`
- **THEN** `showNotes` is updated to `"Summary text"`

#### Scenario: Episode without recap
- **WHEN** regeneration runs for an episode with `recap = null`
- **THEN** `showNotes` is left unchanged

#### Scenario: All existing episodes processed
- **WHEN** regeneration is triggered and 50 episodes exist with recaps
- **THEN** all 50 episodes have their `showNotes` updated to recap-only format
