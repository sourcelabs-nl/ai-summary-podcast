## Context

`TopicDedupFilter.filter()` sees article titles + summaries from the last `recapLookbackEpisodes` (default 7) episodes via `episode_articles` (see `ArticleEligibilityService.findHistoricalArticles`). The stored `episodes.recap` is generated by `EpisodeRecapGenerator` *after* an episode is composed and is never read back into any later pipeline run. `episodes.script_text` is full text but also unused downstream. There is no vector store and no Spring AI tool-calling anywhere in the codebase today. The OpenAI-compatible `ChatClient` is built by `ChatClientFactory.createForModel` and used uniformly across filter/score/compose stages.

This change introduces the first Spring AI tool-calling surface — scoped to the compose stage only — and the database infrastructure to support it.

## Goals / Non-Goals

**Goals:**

- Let the compose LLM ask "was this covered already?" against ALL prior episodes (not just the last 7), via a bounded keyword search.
- Stay within SQLite. No new native deps; FTS5 is sufficient for "did we cover speckit?" keyword queries.
- Lay groundwork that a separate deep-dive-research change can reuse: `ChatClientFactory` compose path, `ToolBudget`.
- Cache safely: tool loops must not break `CachingChatModel`.

**Non-Goals:**

- Semantic search / embeddings. FTS5 is enough for v1; the tool contract is designed so a vector-backed implementation can swap in later without changing callers.
- Replacing `TopicDedupFilter`. The new tool is additive at compose time only.
- Surfacing tool-call traces in the dashboard.

## Decisions

### D1. FTS5 over `recap + script_text + topics`

FTS5 ships in the bundled `sqlite-jdbc`. Indexing `recap` (concise, LLM-generated) gives high-signal hits; `script_text` catches anything the recap missed; the joined `topic` column from `episode_articles` gives clean keyword hooks. `tokenize = 'porter unicode61'` provides English stemming and unicode normalisation.

Alternatives considered:

- Embeddings + `sqlite-vec` — native dep, embedding cost per episode, more moving parts. Deferred; the tool surface is compatible with a later swap.
- LIKE-based search — no ranking, no stemming, poor recall. Rejected.

### D2. Tool surface — `searchPastEpisodes(query) -> List<PastEpisodeMatch>`

Single string parameter. Returns up to 5 matches ranked by `bm25`. Each match: `episodeId`, `generatedAt` (ISO date), `topics` (joined), `recapSnippet` (capped at ~280 chars). No `script_text` field — bounding token usage is more important than completeness.

Scope is per-podcast (the current podcast's id is injected via a per-request factory or constructor; the tool never queries across podcasts).

### D3. ToolBudget wrapper

Spring AI doesn't enforce per-request tool-call caps. We wrap each `ToolCallback` in a `ToolBudget` that counts invocations and short-circuits (returns an empty result with a `budgetExhausted=true` field) once the cap is hit. No exceptions — the compose loop must keep going.

The wrapper is created fresh per compose call. Concurrency safety comes from per-request instantiation; no shared mutable state across episode generations.

Cap for `searchPastEpisodes`: 5 calls per episode.

### D4. ChatClientFactory split

Add a new `createForCompose(userId, resolvedModel, podcast, toolBudget)` that registers tools; keep `createForModel(...)` tool-less for filter/score. This keeps the surface explicit and prevents accidental tool registration for stages that should stay clean.

### D5. Cache key under tool loops

`CachingChatModel` currently keys on `(model, prompt-hash)` and intercepts `call(Prompt)`. Under Spring AI's tool-call loop, the `ChatClient` makes multiple round-trips internally before returning the final `ChatResponse`. We must verify (and assert via a test) that:

- The cache key still uniquely identifies the *user prompt* (not the tool-augmented intermediate messages).
- The cached value is the *final* assistant message (not a partial response prior to tool resolution).
- A second invocation with the same `(model, prompt)` returns the cached value WITHOUT issuing any new tool round-trips.

If the audit reveals drift (e.g. Spring AI re-enters `call(Prompt)` during the tool loop with a modified prompt), we adjust the cache to intercept at a higher level (around the entire `chatClient.prompt().user(prompt).call()` invocation) or hash only the *user* prompt parameter.

### D6. Migration: FTS5 virtual table + triggers + backfill

A single Flyway migration creates:

- `episode_history_fts(episode_id UNINDEXED, podcast_id UNINDEXED, generated_at UNINDEXED, topics, recap, script_text, tokenize='porter unicode61')`.
- Triggers on `episodes` AFTER INSERT and AFTER UPDATE OF `status`, `recap`, `script_text` — insert or replace the FTS row when status is `GENERATED`.
- Triggers on `episode_articles` AFTER INSERT/UPDATE OF `topic`/DELETE — recompute the joined `topics` for the affected episode.
- One-time backfill: `INSERT INTO episode_history_fts ... SELECT ... FROM episodes JOIN episode_articles WHERE status='GENERATED' GROUP BY episode_id`.

Backfill on ~120 episodes (current production size) is instant.

### D7. Prompt nudge

All three composers add the same prompt block (placed near the engagement-techniques section):

> Before treating any topic as new, call `searchPastEpisodes` with the relevant keyword(s). If the tool returns a prior episode that covered the topic, either skip it, treat it as a follow-up referencing the prior coverage, or angle the segment as an update.

This nudge is always present at compose time (not gated by any podcast setting).

## Risks / Trade-offs

- **[Risk] Tool loops break LLM caching.** → Mitigation: regression test asserting second identical compose run is a cache hit with zero tool invocations; falls back to higher-level caching if needed.
- **[Risk] FTS5 backfill is slow on a large episodes table.** → Mitigation: production size is ~120 rows; trivially fast. Migration runs during app startup (offline).
- **[Risk] LLM ignores the tool.** → Mitigation: explicit prompt nudge; verification step in tasks (live run asserts tool was invoked at least once).
- **[Trade-off] Keyword FTS5 misses synonyms ("spec toolkit" vs "speckit").** → Acceptable for v1; tool contract designed for future embedding swap.
- **[Risk] Tool returns stale snippets if recap is updated after publication.** → Mitigation: triggers fire on recap updates.

## Migration Plan

1. Land the FTS5 migration; backfill seeds existing episodes; triggers keep future episodes in sync.
2. Deploy the backend; `ChatClientFactory.createForCompose` registers `searchPastEpisodes` for every podcast immediately. Behavior is strictly additive — the LLM may or may not call the tool, and prior generations are unchanged.

Rollback: revert code; the FTS table can stay or be dropped (no other code depends on it).

## Open Questions

- Should the tool return `script_text` snippets when recap is short? Not in v1 — recap-only keeps token usage bounded. Revisit if the LLM consistently asks "tell me more" via redundant queries.
- Should we log every tool invocation per episode to the database for transparency? Not in v1; `app.log` is enough.
